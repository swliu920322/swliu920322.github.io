搭架自己的微前端

软件设计原则

solid

额外设计原则

  • 组合/聚合复用原则

    • 组合优于继承
    • 23种设计模式

  • 无环依赖

    • A->B->C->A 中介者模式

  • 共同封装

    • 将变化隔离 开放-封闭的拓展

  • 共同重用

  • 好莱坞原则

    • 控制反转（依赖注入）npm包的思想

  • dry原则（不重复）

    • 利用以上的设计原则去做封装

  • 保持简单和傻瓜

    • 保持页面简洁 功能实用 操作方便

  • 高内聚低耦合

  • 关注点分离

  • 你不需要它

    • 架构前期不要过度设计
    • 要足够简单，不失拓展性

    软件设计分层

    系统级架构

  • 定义 应用在这个系统中，如何与后端通信，如何和第三方集成

  • 处理

    • 设计前端
      • 首要：了解前端系统与其他系统之间的关系
      • 包括业务关系和协作机制
    • 设计后端
      • 规定与后端数据通信的机制
      • 包括api设计，token设计，数据传递设计cookie
    • 分离设计
      • 如何实施技术决策，用户鉴权，APi接口管理，Api文档管理，Mock使用，BFF，是否需要服务端渲染

  • 微前端

    • 定义

      • 在一个系统内微前端是应用间的架构方案
      • 在多个应用间，微前端是一种系统间等架构方案，
      • 微前端是将多个前端应用以某种形式结合在一起进行应用。

    • 解决

      • 单体应用在长时间跨度，变成了一个巨石应用后难以维护。

    • 实现形式

      • 单实例，
      • 多实例 同一时刻可以展示多个子应用
        • 通常使用web components方案做子应用封装，子应用更像业务组件而不是应用

      应用级架构

  • 定义 系统级架构的细化

    • 单个应用与其他外部应用的关系，微服务架构下多个应用的协作，数据交换

  • 应用

    • 脚手架 应用生成
    • 模式库 ui库 函数库
    • 设计系统 内部实现的功能，与外部的信息交互

      模块级架构

  • 定义 业务编码前的设计 类似迭代

    代码级架构

  • 定义 规范与原则

    • 开发流程
    • 代码质量与改善
    • 规范而非默契

      注意

    • 开发中，要注意可维护性
    • 简单的代码可维护性高，越抽象越难维护。

如何保证架构的质量-稳定性和健壮性

  • 系统的稳定性

    • 系统处于平衡状态，收到外部影响后，能够回到原来的平衡状态
    • 意义
      • 架构设计的基石
      • 可以更好实现自我修复

  • 系统的健壮性

    • 计算机软件在输入错误，磁盘故障，网络过载或攻击下，不崩溃
    • 容错能力强，运行不易被干扰，安全性好
    • 衡量
      • 从错误的输入推断出正确合理的输入
      • 可以正确运行在不同环境下
      • 检测内部的设计或者编码错误，并且得到正确结果

  • 含义

    • 对特定软件的自身要求
    • 稳定性和健壮性是软件处理的一部分内容
    • 稳定性和健壮性是软件规划时确定的目标

    衡量架构质量

  • 拓展性

  • 维护性

  • 可管理

  • 高可用（故障修复，容灾，降级，熔断）

如何保证架构质量

  • 理解难度，更好理解
  • 接入依赖的成本，易于接入
  • 奔溃率和错误率的指标 低
  • 开发效率 提高开发效率
  • 错误上报和信息收集功能

架构前期准备（正确的选择是良好的开端）

  • 架构师分类
    • 系统架构师 偏向技术
      • 从系统维度，负责整理系统的架构设计
      • 主要是基础服务和各系统间协调，着眼全局
      • 比如关注负载，可靠性，伸缩，扩展，整体项目切分，缓存应用方面的基础架构设计
    • 应用架构师 业务技术并存
      • 从应用程序的维度，负责某个应用的技术架构，偏向业务
      • 关注理解业务，梳理模型，设计模式，接口，数据交互等
      • 了解业务系统的流转
    • 业务架构师 脱离开发
      • 从业务流程维度，关注某行业，业务的领域，获得领域模型，最终获得系统模型
      • 页脚业务领域专家，行业专家，产品咨询师，资深顾问
  • 技术前期准备
    • 技术选型：社区氛围，发展规模，未来发展趋势，与团队契合度，执行成本，维护和迁移成本，执行效率调研和报告
    • 充分调研每一项技术可能的利和弊
    • 最大程度预测架构设计中的缺陷，防止问题发生
  • 技术优化
    • 发展过程中，可能存在有些有悖于当前架构的设计，早餐架构发展阻塞，所以需要进行架构优化，提高适应性
  • 架构优化
    • 不是一蹴而就的，在业务发函中，架构也在不断演进
    • 对架构设计进行实时调休，使架构优化成为常态
    • 通过不断调整，改进不足，补足短板

      技术债务填补和崩溃预防

    • 原因
      • 开发过程中，因为时间紧迫导致实现不合理
        • 算法优化，
      • 暂时没有想好更好的实现方式而妥协
        • if else
      • 架构设计前期没有考虑到的细节
        • 参数传递 -> 状态管理
      • 不合理的交互设计，导致技术实现复杂
      • 旧功能文档缺失，无正确拓展，修改和兼容旧功能，导致上线后问题剧增。
    • 后果
      • 修复变重构 小的技术债务不偿还，最后会演变为大规模的重构
      • 影响开发速度 极大影响上线速度，迭代慢，失去核心竞争力
      • 陷入维护开发的恶性循环
    • 技术填补
      • 思想
        • 优秀的架构设计
        • 有效处理当前需求可预计的情况，
      • 解决方案
        • 根据当前业务，进行项目拆分，尽可能代码解耦合
          • 必须要日志系统，操作日志，错误日志，业务日志
        • 良好的技术培训，和传帮带能力
          • 让每一个开发者可以从更深一层次理解自己实现的功能
          • 从最开始的代码规范，到熟悉业务，最后在编写文档
        • 充分的技术方案
          • 充分理解需求，产生的最理想的实现方案
        • 不同工程师之间相互review代码
          • codeReview也是对自身的一个提高
        • 提升对修复技术债务重要性的认知
          • 遇见一个债务导致的问题，才能花时间去解决
        • 善于发现和定期处理一些技术债务
          • 用于发现系统中的技术债务，让自己对系统负责
      • 总结
        • 上线后，时间不紧，找时间处理技术债务，一遍建立感情和品位原来的代码

预防架构崩溃

  • 系统崩溃的产生
    • 日志记录。操作日志，错误日志，业务日志
  • 预防崩溃
    • 用户行为抓取–争取最新时间获取用户操作链条
    • 解决存量问题-技术债务
    • 遏制新增，减少新增问题的概率
    • 对脏数据进行兜底和检验
    • 单元测试
    • 崩溃报警
    • 自动化测试，整体流程自动化测试
    • 更广的灰度触达
    • 性能优化体系

系统重构

  • 起因
    • 架构不是永恒不变的，也是有生命周期的。
  • 重构概念
    • 对软件内部结构的一种调整
    • 在不改变当前软件的可观察行为的前提下，提高其可理解性，降低修改成本。
  • 实现方式
    • 使用一系列重构手法，调整其结构
    • 理念
      • 运用大量微小的保持软件行为的步骤，一步步达成大规模的修改。
  • 早期系统优势
    • 开发速度快
    • 代码复杂度低
    • 代码规范保持完好
    • 严格注重开发规范，不会允许危及架构设计的代码产生
    • 早期功能简单，拓展起来难度低，成本低
  • 后期系统
    • 具备所有早期其他的劣势
    • 代码复杂度高
    • 代码规范不完善
    • 很多需求和功能出现逾越架构设计的情况
    • 添加新功能兼顾较多，涉及较多模块
  • 流程
    • 确定问题点，确定重构功能和范围
    • 旧架构设计和逻辑梳理

实现方式对比

  1. iframe
  • 优势
    • 技术成熟
    • 支持页面嵌入
    • 天然支持运行沙箱隔离，独立运行
  • 劣势
    • 页面之间可以是不同域名 鉴权
    • 设计一套通讯机制，如何监听，传参
    • 应用加载，渲染，缓存的实现，都在浏览器中难以控制
  1. web component
  • 优势
    • 支持自定义元素，
    • 支持shadow dom，可通过关联进行控制
    • 支持模板template和插槽slot，引入自定义组件
  • 劣势
    • 接入微前端需要重写当前项目
    • 生态系统不完善，技术过新容易出现兼容性
    • 架构设计复杂，组件间拆分过细，通讯控制繁琐
  1. 自研框架
  • 优势
    • 高度定制化，满足兼容的一切场景
    • 独立通信机制和沙箱运行环境，可解决应用之间相互影响的问题
    • 支持不同技术栈的子应用，可无缝实现页面无刷新渲染。
  • 劣势
    • 技术实现难度较高
    • 需要设计一套定制的通信机制
    • 首次加载会出现资源过大的情况
  • 实现
    • 路由分发式
    • 主应用控制路由匹配和子应用加载，共享依赖加载
    • 子应用做功能，接入主应用实现主子应用控制和联动

    • 渲染

  • 微前端框架

    • 子应用的注册
    • 有开始内容（应用加载完成）
    • 路由更新判断
    • 匹配对应的子应用
    • 加载子应用内容
    • 完成所欲依赖项的执行
    • 将子应用渲染在固定容器内
    • 公共事件的管理
    • 异常的捕获和报错
    • 全局状态管理
    • 沙箱隔离
    • 通信机制

  • 服务端功能

    • 提供数据服务

  • 发布平台

    • 主子应用的打包和发布
